# KOB 微服务治理升级方案（服务注册/发现/网关/配置中心）

## 1. 当前现状（基于现有工程结构判断）

当前工程包含 3 个独立服务模块：

- `backend/`：主业务服务（用户、对局、WebSocket、鉴权等）
- `matchingsystem/`：匹配服务
- `botrunningsystem/`：Bot 执行服务

现有通信方式：  
大概率是 `backend` 通过固定地址（如 `http://127.0.0.1:3001`、`3002`）直接调用其它服务。  
这属于“可运行的分布式拆分”，但不具备完整微服务治理能力。

---

## 2. 能不能实现“正常微服务功能”？

可以，且改造成本中等，建议分阶段实施。

你可以实现的核心能力：

- 服务注册与发现（Eureka/Nacos）
- 统一网关（Spring Cloud Gateway）
- 统一配置（Config Server 或 Nacos Config）
- 声明式服务调用（OpenFeign）
- 熔断/限流/降级（Resilience4j + Gateway 限流）
- 链路追踪与日志聚合（Sleuth/Zipkin 或 OpenTelemetry）

---

## 3. 推荐目标架构

前端 -> API Gateway -> backend -> (matchingsystem / botrunningsystem)

其中：

- 所有请求先进入 `gateway`
- `backend` 不再写死 `127.0.0.1:3001/3002`
- `backend` 通过服务名调用（如 `http://matchingsystem/...`）
- 各服务启动时自动注册到注册中心
- 网关层统一做鉴权白名单/CORS/限流/灰度

---

## 4. 分阶段改造路线（推荐）

## 阶段 A：先完成服务注册发现（最关键）

目标：去掉硬编码 IP/端口。

1. 新增注册中心（建议先用 Eureka，简单直观）
2. `backend`、`matchingsystem`、`botrunningsystem` 引入注册客户端依赖
3. 每个服务配置：
    - `spring.application.name`
    - `eureka.client.service-url.defaultZone`
4. 启动后在注册中心页面看到 3 个服务在线

收益：服务实例变更不影响调用方，支持横向扩容。

---

## 阶段 B：引入 OpenFeign 替代手写 HTTP

目标：`backend` 调用其它服务改为声明式接口。

1. 在 `backend` 启用 `@EnableFeignClients`
2. 为匹配服务定义：
    - `@FeignClient(name = "matchingsystem")`
3. 为 Bot 服务定义：
    - `@FeignClient(name = "botrunningsystem")`
4. 删除/替换现有手写 URL 的调用代码

收益：代码更清晰，天然支持服务发现 + 重试/熔断扩展。

---

## 阶段 C：增加网关（Gateway）

目标：统一入口，避免前端直连多个服务。

1. 新建 `gateway` 模块（Spring Cloud Gateway）
2. 在网关中配置路由：
    - `/api/**` -> `backend`
    - （按需）`/matching/**` -> `matchingsystem`
3. CORS、鉴权、限流统一在网关处理
4. 前端 BaseURL 统一改为网关地址

收益：统一安全策略与流量治理，部署更规范。

---

## 阶段 D：配置中心 + 生产治理

目标：让环境切换、密钥、连接串统一管理。

1. 引入配置中心（Nacos Config 或 Spring Cloud Config）
2. 数据库、JWT 密钥、第三方地址外置化
3. 按环境拆分配置：`dev/test/prod`
4. 增加健康检查、指标监控（Actuator + Prometheus）

收益：上线安全性、运维效率、可观测性显著提升。

---

## 5. 对 KOB 项目的特别建议

1. WebSocket 与匹配流程
    - 保持 WebSocket 在 `backend`，匹配结果通过服务调用回传给 `backend`
    - 重点保证：服务间请求超时、重试、幂等处理

2. Bot 执行服务
    - 这是高风险模块，建议单独资源隔离（线程池/容器）
    - 必做超时控制、沙箱限制、防死循环

3. 数据一致性
    - 先保持单库（降低复杂度）
    - 稳定后再考虑按服务拆库

4. 鉴权策略
    - JWT 校验建议在网关 + backend 双层保证
    - 网关做粗粒度，backend 做细粒度权限校验

---

## 6. 最小可行改造（MVP）

如果你想最快落地“像样微服务”，建议只做这 3 件事：

- 上 Eureka（服务注册发现）
- 上 OpenFeign（替换硬编码 URL）
- 上 Gateway（统一入口）

这三步完成后，你的项目就从“手工分布式”升级为“标准微服务基础形态”。

---

## 7. 风险与注意事项

- Spring Boot 与 Spring Cloud 版本必须严格匹配
- 不要一次性重构全部链路，按模块灰度切换
- 先保证本地多进程稳定，再上服务器
- 生产环境必须把数据库与 JWT 密钥改为外部配置，不要硬编码

---

## 8. 结论

结论：**能实现，而且值得实现。**  
你当前项目结构已经具备良好基础，按“注册发现 -> Feign -> 网关 -> 配置中心”的顺序改造，风险最低、收益最大。
